# Organising Team -- Do's and Don'ts

You are the engine of this operating model. You keep submissions flowing, evaluations honest, and the pipeline visible. You're not gatekeepers -- you're operators. Your job is to make it easy for people to submit, fair for people to be evaluated, and clear for everyone to see what's happening.

The organising team should be cross-functional: product, engineering, UX, and at least one person from outside tech. You are the people who are already respected in the org -- the ones others come to for advice. That credibility is what makes this work.

---

## Do's

### 1. Be open-minded about what counts as a prototype

Someone automated a spreadsheet workflow with a macro and an AI API call. That's a prototype. Someone built a Figma flow with a working backend. That's a prototype. Don't define "prototype" so narrowly that only full-stack engineers can participate.

### 2. Use GitHub labels to group similar ideas

Tag submissions by category (ai-ml, automation, data, ux, infrastructure, process). Tag by status (submitted, under-review, selected, invested). Tag by cycle (Q1-2026, Q2-2026). This labelling is how you turn a pile of PRs into a visible pipeline. Be disciplined about it.

### 3. Merge overlapping submissions

If three people independently built a meeting summariser, that's not three competing submissions -- that's one validated problem. Bring those submitters together. Combine the strongest elements. Submit one unified prototype to evaluation. Everyone gets credit.

### 4. Look for patterns in what's being built

If half the submissions in a cycle are about internal search, that's the org screaming about a real problem. Track these patterns. Report them to leadership. The aggregate signal from submissions is often more valuable than any individual prototype.

### 5. Build tooling that helps people submit and prototype

Create Claude Code extensions, templates, starter kits -- anything that lowers the effort to go from idea to working prototype. Front-end design agents, data pipeline scaffolds, API boilerplates. The easier you make it to prototype, the more (and better) submissions you'll get.

### 6. Maintain the dashboard

The dashboard is how everyone -- submitters, leadership, the broader org -- sees what's happening. Keep it current. Update submission data after every review. If the dashboard is stale, people assume the program is stale.

### 7. Be available for coffee chats

Some people have a half-formed idea and just need 15 minutes to talk it through. Make yourself available. These conversations often turn a vague notion into a sharp submission. Approachability is part of the job.

---

## Don'ts

### 1. Don't gatekeep submissions

If someone follows the template and has a working prototype, it gets submitted. You don't pre-screen for "good enough." The rubric and the review panel handle quality assessment. Your job is intake, not filtering.

### 2. Don't add process for process' sake

Every form field, every required document, every approval step you add is friction. Friction kills submissions. Before adding any new requirement, ask: "Does this directly improve evaluation quality?" If the answer is no, don't add it.

### 3. Don't let evaluations pile up

Submissions deserve timely responses. If someone submits in January and doesn't hear anything until the quarterly review in April, that's three months of silence. Acknowledge submissions within a week. Provide preliminary feedback within two weeks. The formal evaluation happens quarterly, but communication doesn't wait.

### 4. Don't play favourites

You know the repeat submitters. You know the senior engineers whose prototypes are always polished. Treat the first-time submitter from the finance team with the same attention and rigour. The rubric is the equaliser -- use it.

### 5. Don't over-engineer the process

This is a GitHub repo with markdown files, labels, and a static dashboard. It works because it's simple. The moment you start talking about custom apps, multi-tenancy, workflow engines, or dedicated infrastructure, you've lost the plot. The overhead of the system should never exceed the value it creates.

### 6. Don't hoard information

Every submission, every evaluation score, every decision should be visible in the repo. Transparency is not optional -- it's the mechanism that builds trust. If people can't see why a submission was selected or rejected, they'll assume the worst.

### 7. Don't forget to close the loop

Every submission needs a final status: selected, not selected (with reason), merged with another submission, or withdrawn. No submission should sit in limbo with an "under-review" label for months. Close the loop, every cycle, for every submission.
